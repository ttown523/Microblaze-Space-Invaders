Changes in 5.00.a introduced in 13.4

13.4 - Changes in VHDL/Verilog/Netlist sources (.vhd, .v, .ngc, .edn) 

New Features:

  • Added optional TUSER(0) frame sync output for MM2S channel and optional TUSER(0) frame sync input for S2MM channel to to be used as start of frame marker.

  • Added frame sync crossbar allowing selection of frame sync sources for each channel when AXI VDMA is configured to use external frame sync (C_USE_FYSNC=1). DMACR bits 5 and 6 were added to allow selection of frame sync sources for each channel.

  • Added optional internal GenLock bus for cases where one channel is configured as a GenLock Master and one channel is configured as a GenLock slave.

  • Expanded number of frame stores from 16 to 32.

Changes in 4.00.a introduced in 13.3

13.3 - Changes in VHDL/Verilog/Netlist sources (.vhd, .v, .ngc, .edn) 

New Features:

  • Added dynamic linebuffer threshold feature allowing users to specify the almost empty and almost full thresholds dynamically via s_axi_lite register write. This feature adds two new registers, MM2S_THRESHOLD (offset 0x1C) and S2MM_THRESHOLD (offset 0x4C) which will default on power up and reset to C_MM2S_LINEBUFFER_THRESH and C_S2MM_LIEBUFFER_THRESH parameters respectively.

  • CR591965 - Added Flush on Frame Sync functionality allowing error'ed frames to be flushed and transfers to restart on subsequent frame sync. This feature is enabled by setting the new parameter C_FLUSH_ON_FSYNC = 1 (default = 1). This feature is only valid when C_USE_FSYNC = 1.

  • Added option to repeat or advance the frame number on frame sync when error-ed frame is detected. Setting C_MM2S_GENLOCK_REPEAT_EN = 1 or C_S2MM_GENLOCK_REPEAT_EN = 1 for the respective channel will cause the error-ed frame to repeat on frame sync. Setting the associated parameter to 0 (default) will cause the frame to advance on frame sync. This feature is only valid for GenLock masters and when C_USE_FSYNC = 1 and C_FLUSH_ON_FSYNC = 1.

  • Added Dynamic Clock Frequency feature. When AXI VDMA is configured for asynchronous operation (C_AXI_IS_ACLK_ASYNC = 1) then clock frequencies are not restricted to build time parameter settings. Also clock frequencies can be changed during operation. Note: Changing clock frequencies during operation will create momentary disruption in video transfers. Enabling flush on frame sync (C_FLUSH_ON_FSYNC = 1) allows AXI VDMA to recover from the clock frequency change.

  • Expanded MM2S and S2MM AXI4 and AXI4Stream data widths to support 512 bit and 1024 bit widths.

Resolved Issues:

  • Modified AXI VDMA to use new AXI Datamover Upsizer/DownSizer allowing throttle-less packing and unpacking of data on MM2S and S2MM AXI4 Interfaces.

  • CR609385 - Removed BUS setting in mpd on reset outputs, mm2s_prmry_reset_out_n and s2mm_prmry_reset_out_n

 

Known Issues / Limitations:

  • No known issues or limitations



 

Changes in 3.01.a introduced in 13.2

13.2 - Changes in VHDL/Verilog/Netlist sources (.vhd, .v, .ngc, .edn) 

New Features:

  • Added dynamic frame store feature allowing users to specify number of frame stores used per channel up to a maximum C_NUM_FSTORES. MM2S_FRMSTORE (offset 0x18) and S2MM_FRMSTORE (offset 0x48) registers were added to support this new feature.

  • CR589994 - Added TCL script process to detect asynchronous clock mode and set/clear C_PRMRY_IS_ACLK_ASYNC automatically.

  • Optimized Video Register Blocks to reduce FPGA resource utilization

  • Added option to disable AXI-Lite reading of video parameters register when AXI VDMA is configured for Register Direct Mode (C_INCLUDE_SG = 0). A new parameter, C_ENABLE_VIDPRMTR_READS, was added to support this feature. Disabling reads will remove the Read MUX from the video parameters and start addresses reducing FPGA resource requirements.

  • Added optional Store-And-Forward feature to MM2S and S2MM channels. This feature allows the MM2S channel to only make read requests on AXI4 interface if there is buffer room available to store the requested data and on S2MM channel to only make write requests if there is enough data in the buffer to complete the write. If LineBuffers are enabled the Store-And-Forward feature will utilize the line buffer for Store-And-Forward.


Resolved Issues:
  • No issues to resolve

  • No known issues or limitations
